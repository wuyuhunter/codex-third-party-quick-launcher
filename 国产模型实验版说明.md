# 国产模型实验版说明

版本：v0.5.0

## 目标

验证 DeepSeek、Qwen、GLM、Kimi、MiniMax 等国产或国内可用模型能否作为 Codex CLI 的模型 provider 使用，并在不影响稳定版安装目录的前提下，提供一个可复测的桌面实验版。

## v0.4 配置结构

从 v0.4.0 起，启动器把模型配置拆成三层；v0.4.1 继续把“模型厂家”改为“模型系列”，并允许一个接入平台同时选择多个模型系列：

- 模型系列：OpenAI / GPT、Qwen / 通义千问、DeepSeek、GLM / 智谱、Kimi / Moonshot、MiniMax、自定义。
- 接入通道：具体的 base_url + KEY，例如 DashScope 官方、DeepSeek 官方、延林、ciii、小蓝。
- 可用模型：每个接入通道只显示自己勾选的模型，并有一个默认模型。

启动页选择接入通道后，模型下拉框只显示该通道可用模型；如果命令行传入错配组合，启动器会自动改为该通道默认模型并打印提示。配置页新增“模型系列 / 此通道可用模型 / 默认模型”，新手不用再从全局模型池里猜模型归属。模力方舟、硅基流动这类聚合平台可以在同一个 provider 下勾选多个模型系列，配置会写入 `vendorIds`，同时保留 `vendorId/vendorName` 兼容旧版本。

## v0.5.0 发布基准

- `v0.5.0` 是当前国产模型适配线的发布基准版本。
- 继续保持 no-config 路线：provider、KEY、adapter 端口和模型参数都不依赖 `~/.codex/config.toml`。
- 保留 Qwen、DeepSeek、GLM、Kimi、MiniMax 的本机 adapter 适配，以及 `yanling`、`ciii_next`、`xiaolan` 对照 provider。
- “检查更新”只保留在“关于”窗口中，不做自动检查，避免普通新手被更新流程打扰。
- 发布包继续排除 `state`、`logs`、`secrets`、`.omx`、`src`、`update`，避免把本机配置、KEY 或 manifest 自指 hash 打进公开包。

## v0.4.1 配置页和推理深度

- 配置窗口默认高度提升到 `920`，最小高度提升到 `840`，服务列表、模型列表和 KEY 列表都加高，避免 KEY 新增、删除、保存按钮被挡住。
- 左侧模型服务提供方列表新增“置顶 / 置底”，保留“上移 / 下移”。
- “模型和推理模式”改为“模型和推理深度”。全局仍保留深度列表，但每个模型现在可以单独配置支持的推理深度，写入 `settings.modelReasoningEfforts`。
- 根据官方文档口径，OpenAI/Codex 的 reasoning effort 属于 `low/medium/high` 这类深度参数；Qwen、GLM、DeepSeek、Kimi、MiniMax 更常见的是模型选择、thinking 开关、thinking budget 或 reasoning_content，不是统一的 low/high/xhigh 深度枚举。因此 v0.4.1 对国产/国内模型默认只开放 `medium`，避免错误暴露不支持的深度；后续如要精确支持，需要在 adapter 里做 vendor-specific thinking 参数转译。

## v0.4.2 配置页修正

- 配置窗口默认高度提升到 `1040`、宽度提升到 `1100`，继续扩大模型服务列表、模型系列列表、可用模型列表和 KEY 列表。
- 模型服务页把“模型系列”和“此通道可用模型”左右并排；“默认模型”改到下面单独一行。
- 模型系列改为复选框列表，选中状态更明确。
- 新建服务默认勾选 OpenAI / GPT；但如果用户把可用模型全部取消，保存结果会保持空列表，不再自动回退到 GPT 模型。
- “模型和推理深度”页签改为“模型系列和版本”，采用“系列 -> 版本 -> 深度”三列结构。模型系列、模型版本、推理深度均支持上移 / 下移；推理深度挂在具体模型版本下。
- 服务配置页的模型系列列表高度提升到 `230`。
- 修复模型系列全部取消后重新勾选时，默认模型下拉不会自动刷新的问题。
- 配置窗口默认宽度提升到 `1250`，最小宽度提升到 `1100`。
- “此通道可用模型”改为复选框列表；勾选模型系列时默认勾选当前系列可用的全部模型，并同步刷新默认模型下拉。
- 修复配置页秒退：空可用模型列表不再触发 fallback 参数绑定错误，隐藏兼容控件也保持多选模式。

## v0.4.3 检查更新

- “检查更新”只放在“关于”窗口里，不再在启动页和配置页直接提示，避免新手频繁接触更新入口。
- 更新源默认从 Gitee raw 读取 `update/manifest.json`，manifest 里统一描述最新版本、下载地址、SHA256 和更新说明。
- 检查到新版本后会先弹确认框，再下载更新包、校验 SHA256，并由独立 updater 覆盖当前安装目录。
- 更新过程保留 `state`、`logs`、`secrets`、`.omx` 等本机数据目录，避免把当前实验环境一起擦掉。

## 兼容方案

- Qwen / DashScope：DashScope 已提供兼容 `/responses` 的接口，实验版通过本机适配器代理到 `https://dashscope.aliyuncs.com/compatible-mode/v1`。
- DeepSeek、GLM、Kimi、MiniMax：这些接口当前主要是 OpenAI-compatible Chat Completions。新版 Codex CLI 已不再支持 `wire_api = "chat"`，因此实验版通过本机适配器把 Codex 的 Responses API 请求转成上游 Chat Completions 请求。MiniMax 当前接入的是 `MiniMax-M2.7` 文本模型，视觉接口不挂入 Codex 主模型选择。
- 本机适配器地址：由启动器动态分配，只绑定 `127.0.0.1`，不作为用户配置项保存。
- Codex provider 定义：启动器只通过本次进程的 `-c model_providers.<id>.*` 参数传入，不写入、不依赖 `~/.codex/config.toml`。
- 适配器脚本：`tools\domestic-responses-adapter.js`。
- 诊断启动入口：`启动 国产模型适配器.cmd`。正常用户不需要手动启动。
- 自动启动：选择 Qwen、DeepSeek、GLM、Kimi、MiniMax 启动 Codex 或点击连通性测试时，实验版会按需自动拉起适配器。
- 模型元数据：选择 Qwen、DeepSeek、GLM、Kimi、MiniMax 时，启动器会自动向 Codex CLI 传入上下文窗口、自动压缩阈值和最大输出 token 配置，避免 `Model metadata ... not found` fallback 提示。
- Transport shim：为彻底消除 Codex TUI 的 unknown model metadata 警告，国产模型启动时 Codex 侧使用内置模型名 `gpt-5.4`，本地适配器再把请求转译到真实上游模型。启动日志会显示 `Transport model: gpt-5.4 -> upstream <真实模型>`。
- 生命周期：WT 中的 wrapper 持有适配器 lease。用户长期不说话不会回收；只有 Codex 进程退出、wrapper 释放 lease 后才会停止适配器。
- 端口冲突：启动器会先校验带随机 nonce 的 `/health` 签名，只复用当前启动器拉起的适配器。如果 `8787` 被其他软件占用，会自动换到 `8788-8799` 或系统分配的空闲端口。
- 连通性测试：测试界面会按 provider 使用对应默认模型，例如 Qwen 使用 `qwen3.6-plus`，DeepSeek 使用 `deepseek-v4-pro`，GLM 使用 `glm-5.1`，Kimi 使用 `kimi-k2.6`，MiniMax 使用 `minimax2.7`；对照 provider 中 `yanling`、`ciii_next`、`OpenAI/小蓝` 使用 `gpt-5.5`。
- 推理档位：实验版按模型动态绑定推理档位。GPT / Codex 类模型可用 `xhigh`；国产模型默认只显示并使用 `medium/high/low`。如果命令行给国产模型传入 `xhigh`，启动器会自动降为该模型允许的默认档位。

## 注意事项

- 这个实验版包含本机运行配置和已预挂 KEY，只能留在本机复测使用，不要作为公开包分发。
- `~/.codex/config.toml` 不是启动器运行态数据库；provider、KEY、adapter 端口都以便携包 state 和进程参数为准。
- 如果从旧终端或旧脚本环境启动便携包，便携目录 marker 优先于 `CODEX_SWITCHER_HOME`，避免误读其他安装目录的 state。
- `glm-5.1` 默认 thinking 需要适配；实验版适配器先关闭 thinking，保证 Codex 看到干净最终回复。
- Kimi K2 模型要求 temperature 约束更严格；实验版适配器按真实上游模型 `kimi-k2.6` 做兼容处理，即使 Codex 侧 transport model 是 `gpt-5.4` 也不会漏判。MiniMax 通过 `minimax2.7` 选择，adapter 会转成上游 `MiniMax-M2.7`，按历史口径使用普通 Chat Completions 参数；为适配 Codex 的多段 system message，adapter 会先合并 system message，并清理 `<think>...</think>` 包裹。
- 如果手动关闭了适配器进程，再次启动 Qwen、DeepSeek、GLM、Kimi、MiniMax 时程序会自动重启；`启动 国产模型适配器.cmd` 仅作为诊断入口保留。
